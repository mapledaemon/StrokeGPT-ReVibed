===========================
StrokeGPT-ReVibed Changelog
===========================

This changelog tracks pull requests made in the StrokeGPT-ReVibed fork, plus
the current unreleased branch only before a pull request exists. Once a branch
has an open PR, its changelog entry should use the PR number instead of
`Unreleased` so follow-up changelog-only PRs are not needed.


2026-04-25 - PR #67 - Extract Motion Feedback Controls Module
--------------------------------------------------------------

Branch: gpt-motion-feedback-controls-split

- Moved motion feedback history rendering and the
  `/motion_feedback_options` save handler out of
  `static/js/motion-control.js` into
  `static/js/motion/feedback-controls.js`. The new module exports
  `configureMotionFeedbackControls` (binds the `renderMotionPatterns`
  callback the auto-disable save handler needs without re-importing
  across module seams), `renderMotionFeedbackHistory` (populates
  `#motion-feedback-history` with the most recent five entries), and
  `saveMotionFeedbackOptions` (POST `/motion_feedback_options` and
  re-renders the catalog when the response carries a fresh
  `motion_patterns` payload).
- Kept `motion-control.js` as the top-level wiring boundary by binding
  the extracted module through `configureMotionFeedbackControls
  ({renderMotionPatterns})` at init, keeping the auto-disable checkbox
  change listener, and re-exporting the three names behind a
  "Compatibility shim - do not extend" marker for older imports.
- Added `/static/js/motion/feedback-controls.js` to the frontend
  aggregation in `tests/_web_support.py` plus a focused seam test in
  `tests/test_web_static_assets.py` that pins the export names, the
  wire-through, and that `renderMotionFeedbackHistory` plus
  `saveMotionFeedbackOptions` no longer live in-line inside
  `motion-control.js`.
- Updated `ROADMAP.md` so the feedback-controls seam is marked as
  split while motion-training editor utilities remain as the next
  low-risk frontend split target.

2026-04-25 - PR #63 - Extract Built-In Pattern Data To JSON
------------------------------------------------------------

Branch: claude-builtin-pattern-json

- Moved the 468-line in-line `PATTERNS = { ... }` block (34 built-in
  motion patterns) out of `strokegpt/motion_patterns.py` and into
  `strokegpt/builtin_patterns.json`. The runtime catalog is now built
  by `motion_patterns._load_builtin_patterns()` running once at import
  time, matching the previous eager-load semantics. Pattern IDs,
  names, action timing, position values, style fields, and
  `MotionPattern.duration_ms` outputs are unchanged.
- Picked JSON over a sibling Python data module because a Python
  module would have to import `MotionPattern`/`PatternAction` from
  `motion_patterns` to construct each entry, while `motion_patterns`
  imports the data back, deadlocking any direct entry into the data
  module. JSON has zero Python imports, so the data file cannot
  participate in a cycle by construction. The dataclasses stay in
  `motion_patterns.py` where every existing caller already imports
  them from.
- Added `tests/test_builtin_patterns.py` with 9 contract tests pinning
  the 34 expected pattern IDs, the minimum action count, the loader's
  fresh-dict semantics, the JSON file's pattern IDs matching loader
  output, the locked endpoint shapes for `stroke` / `milk` /
  `edge-build-low`, the `duration_ms` values for `stroke` (900) and
  `milk` (1280), and a JSON-key allow-list that catches misspellings
  like `windowscale` for `window_scale` before they ship.
- `motion_patterns.py` shrank from 1114 to 691 lines (-423 net).
- Resolved the matching `ROADMAP.md` Up Next #4 entry and renumbered
  the remaining tiers (Up Next 5-8 → 4-7, Queued 9 → 8, items 10-18 →
  9-17), fixed the cross-reference that pointed at the chat-shell
  refactor's old #8 number, and updated the `AGENTS.md` Up Next list
  plus the architecture listing to include the new JSON data file.
